# Decisions — Edgar's Home Automation

*Permanent record of load-bearing decisions that shape how we work. Append-only — never rewrite past entries. Add a new row for reversals or superseded decisions.*

| Date | Decision | Rationale | Alternatives considered |
|------|----------|-----------|------------------------|
| 2026-04-11 | Remote access via HA Cloud is deliberately disconnected (relayer off) | Edgar doesn't need remote access for this setup. HA Cloud stays logged in only for Alexa + Google Assistant integration. | (a) Enable full remote access — not needed; (b) Log out of HA Cloud entirely — would break Alexa/Google. |
| 2026-04-11 | Blackshome motion-light blueprint requires the `dynamic_lighting_boolean` toggle helper per automation for lights that can be switched off at the wall | The blueprint tries to read `brightness` from the target light; if the light is off, the attribute doesn't exist and the automation errors with `UndefinedError: has no attribute 'brightness'`. The `dynamic_lighting_boolean` toggle is the blueprint author's own documented workaround. | (a) Fork the blueprint and fix upstream — higher maintenance cost; (b) Live with the errors — unacceptable since motion light may fail to run. |
| 2026-04-12 | High-frequency monitoring data lives in InfluxDB + Grafana, never in HA recorder via counter helpers | The 2026-04-11 monitoring buildout (16 trigger counters + 7 overrule counters + automation_triggered listener) caused a 20× self-triggering cascade, bloated the SQLite DB from 453→785 MB in <24h, and starved the Green board's 4 GB RAM. HA recorder is not a time-series database. InfluxDB captures every HA state change automatically via the add-on. | (a) Fix the cascade and keep counters — still the wrong data store for frequency; (b) Sample less aggressively — still duplicates data already in InfluxDB. |
| 2026-04-13 | Vacation mode Zocci handling is notify-only, no automated control | La Marzocco API returns HTTP 500 on every `switch.turn_off` call for both `switch.zocci` and `switch.zocci_auto_on_off_ijcbhmz`. `continue_on_error: true` was masking permanent failure, not resilience. `automation.vacation_mode_zocci_warning` notifies Edgar when Zocci turns on during vacation so he can disable the schedule manually in the La Marzocco app. | (a) Retry logic — API never returns success, retries are noise; (b) Remove Zocci involvement entirely — misses the actual risk (schedule-driven turn-on while away). |
| 2026-04-14 | HA Jinja2 templates use bracket notation `states['entity_id']` not dot notation `states.domain.entity` | Dot notation is ambiguous: `states.binary_sensor.foo` may fail or silently return `undefined` in edge cases, especially for entity IDs with certain characters. Bracket notation treats the full entity ID as a single string and is unambiguous. Validated via `ha_eval_template` before automation creation. | Dot notation — works in most cases but has known edge cases; `states()` function — returns string only, cannot access `last_changed`. |
| 2026-04-14 | Agent workflow redesigned: Gate 2 = write code + code review; Gate 3 = standardized deploy pipeline in INSTRUCTIONS.md | Gate 3 critic ran 4 cycles on a simple plan — 2 were false positives from missing HA template knowledge. New `ha-code-reviewer` agent has HA-specific syntax rules baked in. Standardized pipeline removes need to invent and review a deployment plan each time. Old critic files kept as legacy. | (a) Keep original approach/plan review model — too much noise, not enough signal on HA specifics; (b) Remove critic entirely — loses the independent review value. |
| 2026-04-14 | Living Room motion proxy uses both Entrance AND Hall/Upstairs sensors (all-or-nothing) | The Living Room has no dedicated motion sensor. Entrance (`binary_sensor.entrance_motion_sensors`) is geographically closest on the ground floor; Hall (`binary_sensor.eve_motion_20eby9901_occupancy`) covers upstairs corridor. Using both as proxies with global AND logic provides the most conservative shutoff signal — lights only go off if the entire active zone of the house has been quiet for 30 minutes. | (a) Entrance only — misses upstairs movement; (b) Hall only — geographically less relevant for LR; (c) Skip LR from motion check — would turn LR lights off unconditionally. |
| 2026-04-16 | Existing UI-managed automations should be migrated to git-tracked packages when substantively modified | Keeping modified automations in UI-only storage defeats the git-tracked workflow. Migration on edit (not proactively) avoids unnecessary churn. Sequence: write package file → commit → delete UI copy → deploy.sh → verify. | (a) Keep UI-managed forever — no history; (b) Batch-migrate all 24 at once — high risk, low reward for unmodified automations. |
| 2026-04-16 | New automation/helper configs are git-tracked YAML files deployed via SSH (`deploy.sh`), not written live via MCP API | MCP write tools (`ha_config_set_automation` etc.) leave no local history, no diff, no rollback path. Git-tracked packages give full version history, pre-commit validation, and one-command deploy. MCP is retained for reads (entity state, traces, health) and ops (backup, restart). Full CI/CD pipeline ruled out: HA Green is on a private LAN (relayer off), making remote runners unreachable; single-person project with human approval gate makes auto-deploy on push wrong. | (a) Keep pure MCP writes — no history or diff; (b) Full GitOps with self-hosted Actions runner — persistent service, overkill for solo home project; (c) SSH only, no git — loses version history. |
| 2026-04-16 | Gate 1 gains an optional `research-advisor` subagent for unfamiliar design classes — returns a brief with 2–3 alternatives, known anti-patterns, and a recommendation cross-referenced against PROFILE.md + DECISIONS.md, before the main agent commits to an approach | Gate 1 alignment was grounded in the main agent's general knowledge, not ecosystem research. Missed patterns the HA community had already solved (native integrations, vetted blueprints) and risked re-discovering anti-patterns already logged as failures here. Advisor is explicitly not a gate — no verdict, no block — only an opinionated brief the main agent folds into the alignment Edgar sees. Triggered only for novel design classes; skipped for patterns already settled in this table. | (a) Add the research step to `ha-code-reviewer` at Gate 2 — too late, approach is already committed; (b) Integrate obra/superpowers wholesale — generic engineering skills duplicate the domain-specific discipline already encoded in Gates 1–3 and would dilute it; (c) Ad-hoc web search per request — inconsistent, hard to constrain to vetted sources, prone to stale-post bias. |
| 2026-04-22 | Dashboards are **adapted in-place** (extend BubbleDash), not cloned from external templates | Attempted 1:1 clone of jlnbln/My-HA-Dashboard over 9 mapping passes: 239 missing entity references replaced, 14,775 lines of sections-view cruft stripped, 12+ HACS dependencies still required, and ~4,000 lines of `button_card_templates` JavaScript with hardcoded entity IDs that crash with `ButtonCardJSTemplateError` on any missing reference. Even after all errors were cleared, the result remained full of foreign content (Dobby vacuum, Prusa printer, PS5, Anna/Valentin/Simone/Guest person cards) baked too deep in JS to cleanly swap. External dashboards are useful as **visual/UX inspiration** (room-first popups, chip bars, greeting cards) but not as clone-and-adapt starting points. | (a) Clone and adapt (tried, abandoned) — maps cleanly at the entity layer but fails at the content layer where names, icons, and device types are woven into template JS; (b) Fork the upstream repo and track updates — Edgar's HA setup diverges too much to benefit from upstream; (c) Hybrid (clone styling, rewrite content) — ends up equivalent to "adapt BubbleDash with inspiration". |
| 2026-04-22 | The 5-helpers-per-session guideline (INSTRUCTIONS.md §Architectural constraints) is about recorder **write frequency**, not helper count. Exceeding is fine when helpers are low-write. | Reviewer pushed back on `weather_aware_heating.yaml` introducing 9 helpers (8 `input_number` + 1 `input_boolean`) as a likely violation. Edgar clarified: the real concern behind the 2026-04-12 teardown was high-frequency writes causing DB bloat (16 counters × automation_triggered cascades, 453→785 MB in <24h), not raw count. This package adds ~5 writes/day total across all helpers (daily 22:00 run). The two "visibility" helpers (`outdoor_3day_avg`, `heating_offset`) are written 1×/day each — qualitatively different from per-event counters. Logged as a standing clarification: future packages should be evaluated on writes/day, not helper count. | (a) Shrink to 5 helpers via hardcoded base temps — loses UI tunability for rooms Edgar actually adjusts; (b) Move visibility helpers to InfluxDB/Grafana only — adds a second place Edgar has to look for the same data; (c) Keep as-is with explicit recorder impact note in the package header — chosen. |
| 2026-04-22 | Weather-aware heating owns the Wiser Kitchen/LR/Office setpoint at 22:00 daily, base+offset absolute (not delta) | Research showed two paradigms: absolute (system overwrites whatever's currently set with base+offset) vs delta (apply offset to whatever the user most recently set). Absolute chosen for (a) simpler mental model — "at 22:00, my setpoints are always exactly base+offset", (b) no accumulating drift across days, (c) the `base_temp` helpers are UI-tunable so Edgar can still change the effective setpoint via slider, just not via the thermostat UI at 15:00 hoping it sticks past 22:00. Daytime manual overrides survive until the next 22:00 run. Vacation Mode preempts via an `input_boolean.vacation_mode == off` condition. | (a) Delta mode (respect manual overrides forever) — accumulating drift, conflicts with vacation_mode.yaml's own absolute setpoint writes; (b) Delta-only-within-day (reset at midnight) — two systems writing setpoints at different times is fragile; (c) Motion-driven occupancy instead of forecast-driven — Edgar explicitly rejected this scope. |
| 2026-04-24 | FUSION dashboard is **git source-of-truth**: edits go file → commit → MCP `ha_config_set_dashboard(config=...)`. The HA UI "Edit Dashboard" button must NEVER be used on FUSION during the 6-phase build. | FUSION is a storage-mode dashboard that lives in `.storage/lovelace.dashboard_fusion`. The HA UI "Edit Dashboard" button writes directly to that file, bypassing git. A single UI edit silently invalidates the git source — next session's agent reads stale YAML, pushes it back, and overwrites the UI change. This is the same class of failure that killed the jlnbln clone (2026-04-22). FUSION will run for 6 sessions/phases, so the risk compounds over weeks. Solo discipline rule, no tooling enforcement. | (a) Convention-only in plan doc — fragile after the plan doc unloads at session 2+; (b) YAML-mode dashboard via configuration.yaml (no UI edits possible) — ruled out on 2026-04-22 when HA 2026.4 dropped `hui-sections-view` support for YAML mode; (c) Restore from git if divergence detected — detection is manual, reactive. |
| 2026-04-24 | For sidebar nav active-state highlight: prefer **button-card native `state: [{operator: template, value: ...}]`** over `card-mod` CSS templates, even though FUSION-DESIGN-SPEC §4 suggests card-mod. | Button-card has the primitive built in — one `state:` block per icon, JS template returning bool, styles applied when true. card-mod achieves the same result via CSS-level state attribute targeting, but adds a layer that must be debugged when styles don't apply. Native mechanism = fewer moving parts, clearer diff when reviewing, works without card-mod loaded. Use card-mod only where the need is actually CSS (selectors into shadow DOM, pseudo-elements, keyframes). | (a) card-mod per the spec — spec was design, not implementation; (b) class-based highlighting via `styles` + `state` mapping (enum operator) — button-card supports this but the template form handles the case uniformly across all 7 icons with one pattern. |
| 2026-04-24 | For panel-switcher content in FUSION, **use 7 parallel `conditional` cards**, not `custom:config-template-card` with `${VAR}` substitution. | config-template-card evaluates nested string values as JS template literals. Any content containing backticks (e.g. markdown code-spans: `` `FUSION-DESIGN-SPEC.md` ``) closes the implicit template literal mid-string, producing "Unexpected token" at re-render time. The 7-conditional-card fallback was already in the plan as R3 mitigation; the pivot took <10 minutes once confirmed. Conditional cards use HA built-ins, no JS eval, and support arbitrary markdown content. Keep config-template-card available for Phase 2+ cases where dynamic card types (not just content) are needed — but check for backticks in nested strings before using. | (a) Keep config-template-card, strip backticks from all content — fragile, any future contributor could add backticks and break it silently; (b) Escape `${...}` pattern — works but obscure; (c) Abandon config-template-card entirely — premature; Phase 2+ may still benefit for dynamic `type:` substitution. |
